# UX Redesign Plan — 제조기술 로드맵 인사이트보드

> 목적: 첨부 구조도(데이터 입력 → 저장·정제 → SOLA 분석 → 서비스 UI → 최종 산출물)를 기준으로, 현재 Streamlit UI/UX를 사용자가 따라가기 쉬운 업무 흐름형 인사이트보드로 재구성한다.
>
> Codex가 UI 개편 작업을 진행할 때 이 문서를 기준 문서로 삼고, 실제 구현은 작은 PR 단위로 나누어 진행한다.

## 1. 개편 배경

현재 시스템은 뉴스 수집, 로드맵 업로드, 트렌드 분석, SOLA 요약/제안, 북마크 등 핵심 기능을 갖추고 있다. 그러나 UI가 기능 단위로 흩어져 있어 사용자는 다음을 판단하기 어렵다.

- 처음 들어오면 무엇부터 해야 하는가?
- 뉴스 수집, 로드맵 업로드, 인사이트보드, SOLA 작업실의 순서는 무엇인가?
- SOLA는 별도 채팅 기능인가, 현재 화면을 도와주는 분석 엔진인가?
- 생성된 인사이트를 어떻게 자동화 과제나 제안서로 전환하는가?

개편의 핵심은 기능을 더 추가하는 것이 아니라, **사용자가 매일 반복할 수 있는 업무 시나리오**로 화면을 재배치하는 것이다.

## 2. 목표 제품 이미지

> 제조기술 로드맵과 외부 뉴스 데이터를 결합해, 부서별 기술 트렌드와 실행 가능한 자동화 과제를 한 화면에서 도출하는 AI 인사이트보드.

사용자는 앱에 들어와 다음 4단계를 자연스럽게 수행할 수 있어야 한다.

1. 내 역할과 관심 공정을 설정한다.
2. 뉴스/로드맵/LLM 데이터 준비 상태를 확인한다.
3. 오늘의 핵심 트렌드와 내 부서 관련 인사이트를 본다.
4. 자동화 과제 후보를 검토하고 SOLA로 제안서 초안을 만든다.

## 3. 현재 UI/UX 문제 진단

### 3.1 기능 단위 네비게이션

현재 구조는 `홈 · 탐색 · 작업실` 아래에 기능 탭이 배치되어 있다. 개발자에게는 모듈 경계가 명확하지만, 사용자에게는 업무 순서가 보이지 않는다.

문제 예시:

- `탐색` 안에 뉴스 수집, 로드맵 업로드, 인사이트보드가 함께 있어 “탐색”의 목적이 모호하다.
- `작업실` 안에 SOLA, 제안서 작업장, 뉴스 콘텐츠, 북마크가 함께 있어 “산출물 관리”와 “AI 작업”이 섞인다.
- 첫 화면에서 다음 행동이 명확하지 않다.

### 3.2 구조도의 5단계 흐름이 화면에 드러나지 않음

첨부 구조도는 다음 흐름을 갖는다.

1. 데이터 입력
2. 저장·정제
3. AI 분석 엔진(SOLA)
4. 서비스 UI
5. 최종 산출물·활용

현재 UI는 이 흐름이 명시적으로 보이지 않는다. 사용자가 “내가 지금 5단계 중 어디에 있는지” 알기 어렵다.

### 3.3 페르소나 설정의 중요도가 낮게 보임

이 시스템의 핵심 가치는 부서/직무/관심 공정 기반 맞춤 인사이트다. 따라서 페르소나는 단순 사이드바 설정이 아니라 첫 사용 경험의 핵심 입력값이어야 한다.

### 3.4 SOLA의 정체성이 분산됨

SOLA는 홈 인사이트, 인사이트보드 해석, 사이드 채팅, 요약, 제안서 생성 등에 흩어져 있다. 개편 후 SOLA는 별도 탭이기보다 **모든 화면에 붙는 AI 분석/산출물 생성 엔진**으로 인식되어야 한다.

### 3.5 디자인은 깔끔하지만 제품 콘셉트가 약함

현재 디자인 토큰은 화이트/그레이/블루 기반으로 정돈되어 있다. 하지만 목표 이미지는 단순 Streamlit 앱이 아니라 **B2B 제조기술 의사결정 대시보드**다. 정보 위계, 상태 카드, 추천 행동, 산출물 CTA가 더 강하게 드러나야 한다.

## 4. 새 정보 구조(IA)

기존 `홈 · 탐색 · 작업실` 구조를 아래 5개 1차 메뉴로 재구성한다.

| 새 메뉴 | 역할 | 기존 기능 매핑 |
|---|---|---|
| 오늘의 보드 | 매일 첫 진입 화면. 핵심 변화, 내 부서 인사이트, 추천 행동 표시 | 홈 + 인사이트보드 핵심 |
| 데이터 관리 | 뉴스 수집, 본문 enrich, 로드맵 업로드, 데이터 품질 상태 | 뉴스 수집·Enrich + 로드맵 업로드 |
| 인사이트 분석 | 트렌드, 뉴스-로드맵 매칭, 자동화 기회 매트릭스 | 인사이트보드 |
| SOLA 작업실 | 요약, 과제 후보, 제안서 초안, 자유 질의 | SOLA + 제안서 작업장 |
| 산출물 보관함 | 북마크, 채택 과제, 제안서, 히스토리 | 북마크 + 뉴스 콘텐츠 일부 |

### 최종 메뉴 트리

```text
오늘의 보드
├─ 오늘의 핵심 변화
├─ 내 부서 맞춤 인사이트
├─ 자동화 기회 Top 5
└─ 추천 다음 행동

데이터 관리
├─ 데이터 준비 상태
├─ 뉴스 수집
├─ 본문 Enrich
├─ 로드맵 업로드
└─ 데이터 품질 점검

인사이트 분석
├─ 기술 트렌드
├─ 뉴스-로드맵 매칭
├─ 자동화 기회 매트릭스
├─ 부서별 인사이트
└─ 상세 필터/드릴다운

SOLA 작업실
├─ 뉴스 브리핑
├─ 과제 후보 생성
├─ 제안서 초안 생성
└─ 자유 질의

산출물 보관함
├─ 북마크 뉴스
├─ 채택 과제
├─ 제안서
└─ 최근 대화/히스토리
```

## 5. 추천 전체 레이아웃

```text
┌─────────────────────────────────────────────────────────────┐
│ 상단 헤더: 제조기술 로드맵 인사이트보드 | 데이터 상태 | SOLA │
├──────────────┬──────────────────────────────────┬───────────┤
│ 좌측 내비     │ 메인 콘텐츠                       │ SOLA 패널 │
│              │                                  │           │
│ 오늘의 보드   │ 현재 화면의 핵심 카드/차트/리스트    │ 질문/요약  │
│ 데이터 관리   │                                  │ 제안 생성  │
│ 인사이트 분석 │                                  │           │
│ SOLA 작업실   │                                  │           │
│ 산출물 보관함 │                                  │           │
└──────────────┴──────────────────────────────────┴───────────┘
```

원칙:

- 좌측은 현재 업무 단계와 목적을 보여준다.
- 중앙은 카드/차트/리스트 중심의 메인 콘텐츠다.
- 우측 SOLA 패널은 현재 화면 컨텍스트, 페르소나, 채택 제안서를 자동으로 이해하는 조력자다.
- 화면마다 primary CTA는 하나만 둔다.

## 6. 화면별 개편 계획

### 6.1 오늘의 보드

첫 화면은 “무엇을 해야 할지 모르는 문제”를 해결해야 한다.

구성:

```text
[Hero]
제조기술 로드맵 인사이트보드
오늘 주목할 기술 변화와 실행 가능한 자동화 과제를 확인하세요.

[상태 카드 4개]
- 오늘 수집 뉴스
- Enrich 완료율
- 로드맵 작업 수
- SOLA 상태

[오늘의 추천 행동]
1. 용접 로봇 관련 뉴스 12건 증가 → 인사이트 보기
2. 내 부서 관련 자동화 과제 3건 → 과제 검토
3. 로드맵 미연결 뉴스 8건 → 매칭 확인

[트렌드 펄스]
- 새 키워드
- 상승 키워드
- 사라진 키워드

[부서 맞춤 인사이트]
- 내 부서 관련 뉴스
- 관련 작업
- SOLA 한 줄 해석

[자동화 기회 Top 5]
- 영향도
- 실행 가능성
- 관련 뉴스
- 제안서 생성 버튼
```

완료 기준:

- 사용자가 첫 화면에서 오늘의 변화, 내 부서 관련성, 다음 행동을 즉시 이해한다.
- 빈 상태에서도 “페르소나 설정 → 데이터 준비 → 인사이트 확인” 순서가 보인다.

### 6.2 데이터 관리

데이터 관리는 설정 폼이 아니라 “데이터 준비 상태 관리” 화면이어야 한다.

구성:

```text
[데이터 준비 상태]
뉴스 DB: 정상 / 오늘 N건
로드맵 DB: 정상 / 작업 N개
Enrich: N% 완료
LLM: 연결됨 또는 미설정

[1. 뉴스 수집]
- 추천 키워드 칩
- 직접 키워드 입력
- 소스 선택
- 수집 실행

[2. 본문 Enrich]
- 미처리 기사 수
- Enrich 실행
- 실패 목록

[3. 로드맵 업로드]
- 현재 로드맵 파일 정보
- 새 파일 업로드
- 스키마 검증 결과

[4. 데이터 품질]
- 중복 기사
- 본문 누락
- 키워드 누락
- 로드맵 매칭 불가 뉴스
```

완료 기준:

- 사용자가 데이터가 준비됐는지 한눈에 파악한다.
- 수집, enrich, 로드맵 업로드의 순서가 명확하다.
- 실패 시 다음 조치가 표시된다.

### 6.3 인사이트 분석

인사이트 분석은 차트와 표 나열이 아니라 분석 스토리로 구성한다.

흐름:

```text
1. 지금 뜨는 기술은 무엇인가?
   - 트렌드 카드
   - 새/상승/사라진 키워드
   - 일자별 볼륨

2. 우리 로드맵과 어디에 연결되는가?
   - 부서 / 공정 / 작업별 매칭
   - 관련 뉴스 묶음

3. 어떤 자동화 과제가 가능한가?
   - 영향도 x 실행가능성 매트릭스
   - Top opportunity

4. 무엇을 실행할 것인가?
   - 과제 후보 선택
   - SOLA에게 제안서 생성 요청
```

완료 기준:

- 사용자가 `트렌드 발견 → 로드맵 연결 → 자동화 기회 도출 → 제안서 생성` 흐름을 따라간다.
- 매트릭스와 상세 매칭 표가 같은 의사결정 흐름 안에 배치된다.

### 6.4 SOLA 작업실

SOLA는 채팅창이 아니라 산출물 생성 엔진으로 보여야 한다.

구성:

```text
[SOLA에게 시킬 작업을 선택하세요]

카드 1. 뉴스 브리핑 만들기
- 선택한 뉴스 기반 요약
- 부서별 영향
- 핵심 키워드

카드 2. 자동화 과제 후보 만들기
- 관련 뉴스
- 연결 작업
- 기대효과
- 난이도

카드 3. 제안서 초안 만들기
- 배경
- 문제 정의
- 적용 기술
- 기대효과
- 실행 계획

카드 4. 자유 질의
- 현재 화면 컨텍스트
- 내 페르소나
- 채택 제안서
```

완료 기준:

- 사용자가 SOLA에게 무엇을 시킬 수 있는지 카드만 보고 이해한다.
- 결과물이 저장, 북마크, 제안서 작업장, 산출물 보관함으로 자연스럽게 이어진다.

### 6.5 산출물 보관함

북마크, 채택 과제, 생성 제안서, 최근 대화를 한곳에서 관리한다.

구성:

```text
[저장된 인사이트]
- 북마크 뉴스
- 채택한 자동화 과제
- 생성한 제안서
- 최근 SOLA 대화

[필터]
- 부서
- 공정
- 기술 키워드
- 상태: 검토중 / 채택 / 보류

[액션]
- 제안서 다시 열기
- Markdown 다운로드
- 후속 질문
- 관련 뉴스 보기
```

완료 기준:

- 사용자가 “읽은 것”, “검토할 것”, “작성한 것”을 한곳에서 찾는다.
- 산출물이 일회성 결과가 아니라 지속 관리되는 자산처럼 보인다.

## 7. 디자인 방향

### 7.1 제품 콘셉트

**Clean B2B Intelligence Dashboard**

조선·제조 도메인의 신뢰감은 유지하되, 복잡한 ERP 느낌이 아니라 최신 AI SaaS 대시보드처럼 가볍고 명확하게 구성한다.

### 7.2 디자인 키워드

- Navy + Teal + Blue
  - Navy: 제조기술/신뢰감
  - Teal: 데이터/자동화
  - Blue: AI/SOLA
- 카드 기반 레이아웃
- 상태 중심 UI
- 큰 숫자 + 짧은 문장
- 우측 SOLA AI 패널
- 화면당 primary CTA 하나

### 7.3 색상 토큰 제안

```css
--navy-900: #0B1F4D;
--navy-700: #123875;
--blue-600: #2563EB;
--teal-600: #0D9488;
--cyan-50: #ECFEFF;
--slate-50: #F8FAFC;
--slate-900: #0F172A;

--status-ok: #16A34A;
--status-warn: #D97706;
--status-danger: #DC2626;
--status-info: #2563EB;
```

주의:

- 배경은 거의 흰색/옅은 slate로 유지한다.
- 카드와 표는 흰색 표면을 기본으로 한다.
- navy/blue/teal은 강조와 상태에만 제한적으로 쓴다.
- 색보다 여백, 정렬, 정보 위계를 우선한다.

### 7.4 공통 컴포넌트 후보

신규 또는 확장 후보:

- `MetricCard`: 큰 숫자와 변화량
- `StatusCard`: 정상/주의/필요 조치 상태
- `InsightCard`: SOLA 요약/인사이트 문장
- `ActionCard`: 추천 행동 + CTA
- `EmptyState`: 데이터 없음/설정 필요/LLM 미설정 안내
- `StepGuide`: 1→2→3 진행 단계
- `FilterBar`: 부서/공정/기간/키워드 필터
- `OpportunityCard`: 자동화 과제 후보

## 8. 사용자 시나리오

### 8.1 처음 사용하는 사용자

```text
1. 앱 진입
2. 시작하기 온보딩 표시
3. 부서 / 직무 / 관심 공정 설정
```

## 9. 단계별 구현 로드맵

### Phase 0. UX 기준 확정

목표:

- 이 문서를 기준으로 새 IA, 화면 목적, 디자인 방향을 확정한다.

작업:

- 현재 메뉴/탭/기능 목록화
- 중복 기능 정리
- 화면별 역할 정의
- 필요한 공통 컴포넌트 목록 확정

산출물:

- `docs/UX_REDESIGN_PLAN.md`
- 필요 시 간단한 와이어프레임 이미지 또는 Mermaid 다이어그램

### Phase 1. 앱 쉘/네비게이션 개편

목표:

- 탭 중첩을 줄이고 업무 단계 중심 메뉴로 바꾼다.

작업:

- `app.py` 라우팅 변경
- `ui/sidebar.py` 메뉴명 변경
- `홈 · 탐색 · 작업실`을 5개 업무 메뉴로 재구성
- 우측 SOLA 패널의 의미를 모든 화면에서 통일

완료 기준:

- 좌측 메뉴만 보고 기능 목적을 이해할 수 있다.
- 첫 진입은 `오늘의 보드`다.

### Phase 2. 디자인 시스템 정리

목표:

- 카드, 버튼, 상태, 빈 화면, 필터를 일관된 문법으로 통일한다.

작업:

- `assets/styles.css` 토큰 정리
- `ui/styles.py` 또는 신규 `ui/components.py`에 공통 컴포넌트 추가
- 기존 화면의 inline style 점진 제거

완료 기준:

- 모든 화면의 헤더, 카드, 버튼, 필터, 빈 상태가 같은 언어로 보인다.

### Phase 3. 오늘의 보드 재설계

목표:

- 첫 화면에서 오늘의 변화와 추천 행동을 보여준다.

작업:

- Hero 영역
- 데이터 상태 카드
- 추천 행동 카드
- 트렌드 펄스
- 부서 맞춤 인사이트
- 자동화 기회 Top 5

완료 기준:

- 사용자가 첫 화면만 보고 다음 행동을 결정할 수 있다.

### Phase 4. 데이터 관리 통합

목표:

- 수집, enrich, 로드맵 업로드를 데이터 준비 상태 중심으로 재배치한다.

작업:

- 뉴스 DB/로드맵 DB/Enrich/LLM 상태 카드
- 뉴스 수집 플로우 단순화
- 로드맵 업로드 통합
- 데이터 품질 점검 섹션 추가

완료 기준:

- 데이터 준비 여부와 필요한 조치가 한눈에 보인다.

### Phase 5. 인사이트 분석 재설계

목표:

- `트렌드 → 로드맵 연결 → 자동화 기회 → 제안서` 흐름을 만든다.

작업:

- 트렌드 섹션 재배치
- 매칭 필터 상단 고정
- 자동화 기회 카드/매트릭스 개선
- 기회 카드에서 SOLA 제안서 생성으로 연결

완료 기준:

- 사용자가 분석 결과를 실행 과제로 전환할 수 있다.

### Phase 6. SOLA 작업실/산출물 보관함 정리

목표:

- SOLA는 산출물 생성 엔진, 보관함은 결과물 관리 장소로 분리한다.

작업:

- SOLA 작업 유형 카드화
- 생성 결과 저장/채택/다운로드 플로우 정리
- 북마크/제안서/채택 과제 통합 보기

완료 기준:

- SOLA 결과물이 일회성 답변이 아니라 관리 가능한 산출물로 남는다.

## 10. 구현 시 주의사항

- 한 PR에서 앱 전체를 갈아엎지 않는다. `app.py`/`sidebar.py`/각 탭 개편은 단계별로 나눈다.
- 기존 테스트를 깨지 않도록 데이터 처리 로직과 UI 레이아웃 변경을 분리한다.
- `on_click=` 금지, pending flag + `st.rerun()` 패턴을 유지한다.
- 외부 HTTP는 `scraping.http.build_session()` 진입점을 유지한다.
- 사용자/외부 문자열을 `unsafe_allow_html=True`로 렌더할 때는 `html.escape()`를 유지한다.
- 문서 개편 후 실제 구현 PR마다 `CHANGELOG.md`와 `docs/SESSIONS.md`를 갱신한다.

## 11. Codex 작업 추천 순서

1. **문서 확정 PR**: 이 문서를 추가하고 팀/사용자 피드백 반영.
2. **앱 쉘 PR**: 메뉴명과 라우팅을 5개 업무 메뉴로 변경.
3. **디자인 시스템 PR**: 공통 카드/상태/빈 화면 컴포넌트 추가.
4. **오늘의 보드 PR**: 첫 화면을 추천 행동 중심으로 재설계.
5. **데이터 관리 PR**: 뉴스/로드맵/enrich 상태 중심으로 통합.
6. **인사이트 분석 PR**: 트렌드→매칭→기회→제안 흐름으로 재배치.
7. **SOLA/보관함 PR**: 작업 유형 카드와 결과물 관리 흐름 정리.
